## For the Project Designer (Human Review)

This is a documentation and scaffolding milestone, so there's no hardware to test yet.
The goal is to verify that everything is in place before functional implementation begins.

### Safety Documentation

- [ ] **Read SAFETY.md.** This is the most important file in the repo. Does it accurately
  describe the risks of e-stim? Does it cover the contraindications you'd expect
  (pacemakers, pregnancy, above-the-waist placement)?
- [ ] **Emergency stop section in SAFETY.md** — does the description of what `stopAll()`
  does make sense to you? Key point: BLE disconnect stops the device immediately
  (no relay layer), unlike the WS provider.
- [ ] **BLE disconnect behavior** — confirm you understand the safety difference: if
  the Bluetooth connection drops, the Coyote stops immediately. This is safer than
  the WS variant.

### Controls Documentation

- [ ] **Read CONTROLS.md.** Does the panel layout diagram match your expectations for
  what the PlayRooms UI should look like for this device?
- [ ] **Channel A and B** — are the ramp slider (0–200) and waveform picker controls
  documented as you'd expect?
- [ ] **Emergency stop button** — is it described as always visible and always active?
- [ ] **Physical dial indicators** — do you understand that Device Dial A/B are read-only
  displays of the hardware scroll wheel, not controls?

### README

- [ ] **Read README.md.** Does the BLE vs WS comparison table accurately reflect
  the tradeoffs you understand between the two providers?
- [ ] **Host setup section** — does the HA add-on configuration (`host_dbus: true`,
  `uart: true`, `usb: true`) match what your setup needs?
- [ ] **Standalone Docker section** — does the `--privileged` approach look correct
  for your Docker environment?
- [ ] **Safety warning** — is it prominent enough at the top of the README?

### Repository Structure

- [ ] Run `ls` in the repo root and verify these files exist:
  `manifest.yaml`, `SAFETY.md`, `CONTROLS.md`, `CHANGELOG.md`, `NOTICE.md`,
  `README.md`, `package.json`, `tsconfig.json`, `src/index.ts`,
  `qa/v1.0.0-dglab-ble-scaffold.md`
- [ ] **CHANGELOG.md** — does v1.0.0 entry accurately describe what was created
  in this milestone?

### One Physical Safety Check (even though no code runs yet)

- [ ] **Confirm understanding of electrode placement:** Before any testing with real
  hardware happens in a later milestone, confirm that you have read and understood
  the "never above the waist" rule in SAFETY.md. This cannot be verified by software.

### src/index.ts — Structural Check

- `PROVIDER_NAME = "dglab-ble"`

- B1 characteristic (notify, scroll wheel feedback)
- BF characteristic (soft limits, frequency balance)
- Battery service `0x1500`
- Emergency stop (zero both channels)
- BLE disconnect/reconnect handling

### manifest.yaml — Field Validation

Verify:
- `name: dglab-ble`
- `type: device-provider`
- `providerApiVersion: 1`
- `requirements.bluetooth: true`
- `requirements.serial: false`
- `requirements.network: false`
- `aiInteraction.allowed: true` and `defaultEnabled: false`
- `riskFlags` contains `electrical-stimulation` with `severity: high`
- `riskFlags` contains `requires-physical-safety-gear` with `severity: medium`
- No `network-dependent` risk flag (BLE disconnect is a hardware failsafe — intentionally absent)
- `deviceLimits.maxCommandRate: 10`
- `deviceLimits.coalescingStrategy: "latest-wins"`
- `settings.scanTimeout.default: 15000`

### SAFETY.md — Content Accuracy Check

Verify these sections exist and contain accurate information:

1. **Absolute Contraindications** — must include pacemakers/ICDs, pregnancy, epilepsy
2. **Electrode Placement** — must state "NEVER above the waist" as the primary rule
3. **Intensity Ranges** — must document 0–200 range with physical meaning
4. **Emergency Stop Behavior** — must describe:
   - Mode flags `0b11` (absolute) for both channels
   - Intensity 0 for both channels A and B
   - Bypasses queuing/rate limiting
   - < 100ms latency when device is connected
5. **BLE Disconnect Behavior** — must document that device stops immediately on
   BLE disconnect (hardware failsafe), and contrast with WS provider behavior
6. **B0 Keepalive Safety Property** — must document that device stops if B0 commands
   stop arriving (~200ms timeout)
7. **Physical Failsafes** — must mention physical power button access and electrode removal
8. **For Developers section** — must list the conditions requiring SAFETY.md update

### CONTROLS.md — Schema Accuracy Check

Verify panel controls match the expected schema:
- Channel A intensity: `rampSlider`, range 0–200, guest ramp mandatory
- Channel B intensity: same as A
- Channel A waveform: `patternPicker`
- Channel B waveform: `patternPicker`
- Linked group toggle for A+B sync
- Emergency stop button — documented as always visible and always active
- Battery indicator — reads from BLE battery service UUID `0x1500`
- Device Dial A/B — read-only, sourced from B1 characteristic notify

Verify B0 command structure section documents:
- 20-byte payload
- Mode flags in first 2 bytes
- Channel A intensity (10-bit)
- Channel B intensity (10-bit)
- 4 waveform samples

### Emergency Stop — Documentation Consistency Check

Verify emergency stop behavior is described **consistently** across all three
documents:

  stop behavior implicitly

No contradictions should exist between these three sources.
